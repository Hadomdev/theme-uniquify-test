# Унікалізація теми — документ рішення

## 1. Що таке "унікалізація теми" з точки зору Google

Google не порівнює сайти "на око". Він шукає **footprint (відбиток) мережі** — технічні збіги, які видають спільне походження:

- однакова **HTML-розмітка** і структура шаблонів (хедер, тіло, футер);
- однакові **імена CSS-класів** і `id` (`.inside-article`, `generate-*` тощо);
- однакові **шляхи до ассетів** (`/wp-content/themes/<one-theme>/...`), однакові хеші/версії файлів;
- однаковий `generator`-мета, однаковий набір плагінів, однаковий хостинг/IP/DNS, однакові аналітика та шаблони внутрішньої перелінковки.

Тобто унікалізація — це **не "інший колір"**, а зменшення кількості цих спільних сигналів, щоб краулер не зміг впевнено зв'язати сайти в один кластер.

## 2. Шари унікалізації (за важливістю)

1. **Структурний / HTML (найважливіший).** DOM, розмітка шаблонів, імена класів, наявність/відсутність сайдбара, `body_class`. Два сайти можуть виглядати по-різному, але мати **ідентичний скелет** — і це головний сигнал мережі. Найдорожчий і найцінніший шар.
2. **Функціональний.** Різні плагіни, різні структури кастомних полів (ACF), різний функціонал. Footprint на рівні поведінки/коду.
3. **Технічний-мета.** `style.css` headers, прибраний `generator`, різні версії/шляхи ассетів, відсутність спільних коментарів.
4. **Візуальний (найлегший, але недостатній сам по собі).** Палітра, типографіка, layout.

**Найважливіший — структурний.** Колір і шрифт міняються за хвилини й нічого не вирішують, якщо HTML однаковий.

## 3. Як організувати: 5 сайтів за тиждень

- **День 0:** зібрати **чек-лист уникалізації** + **бібліотеку варіантів** (3–4 варіанти хедера, футера, палітр, шрифтових пар, layout'ів) і **скрипт-скаффолдер** дочірніх тем.
- **Ключове рішення:** не робити всі 5 сайтів на **одному** батьківському — взяти **2–3 різні parents** (GeneratePress, Astra, Blocksy). Це найсильніший удар по footprint.
- Кожному сайту — свій набір "дизайн-токенів" + свій варіант структури з бібліотеки (без повторів).
- **Фінал:** аудит кожного сайту на залишковий відбиток (порівняти DOM/класи/ассети попарно).

## 4. Що спочатку / в кінці

- **Спочатку:** структурна диференціація (розмітка, класи, layout) + чистка мети та ассетів — це найдорожче й визначає результат.
- **В кінці:** візуальний полішинг (відтінки, відступи, дрібниці).

## 5. Ризики та мінімізація

| Ризик | Мінімізація |
|---|---|
| Залишковий спільний відбиток (спільний parent) | варіювати батьківські теми; переписувати розмітку тіла, а не лише обгортку |
| Авто-генерація створює **новий** патерн | рандомізувати варіанти, ручний фінальний аудит |
| Поломка функціоналу при переписуванні шаблонів | тестувати кожен сайт; не чіпати логіку, лише розмітку/класи |
| Брак часу | пріоритет структури над красою |

## Що реалізовано в цьому завданні

Дві дочірні теми GeneratePress, які виглядають і влаштовані як **різні продукти**:

- **MediClinic Care** (`theme-a`) — мед-клініка: беж+зелений, Fraunces/Inter, лого по центру + меню під ним, 4-колонковий футер, layout з **правим сайдбаром**, класи `mc-*`, ACF-група **Clinic Info**.
- **NovaSaaS Studio** (`theme-b`) — SaaS: темний+фіолет, Space Grotesk/IBM Plex Sans, лого зліва + меню+CTA справа, 2-колонковий футер, layout **без сайдбара** (full-width), класи `ns-*`, ACF-група **Pricing Highlight**.

Різниця не лише косметична: **різна структура тіла** (сайдбар vs його відсутність → різний `body_class` і DOM), різні класи статті (`.mc-article` vs `.ns-entry`), повністю переписані хедер/футер через хуки GeneratePress (`remove_action('generate_header'…)`), прибраний `generator`-мета.

**Чесне обмеження підходу:** оскільки обидві теми — дочірні **одного** GeneratePress, вони все ще ділять частину відбитка батька (базова розмітка циклу, шляхи ассетів parent'а). У реальній мережі це добивається **різними parents** для різних сайтів — про що сказано в п.3. Тут це свідома демонстрація структурного шару в межах одного parent'а.
